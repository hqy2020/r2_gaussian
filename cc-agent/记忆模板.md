# Neo4j Memory 记忆模板

本文档提供 4 种记忆类型的标准模板，确保记忆的颗粒度和结构符合最佳实践。

---

## 📋 模板使用指南

### 基本原则

1. **单个 observation < 150 字**（中文约 100 字）
2. **每个 memory 最多 3 个 observations**
3. **保持类型纯度**（不混合多种类型）
4. **原子化知识点**（可独立理解和复用）

### 何时使用哪种类型

| 类型 | 适用场景 | 关键词 |
|------|----------|--------|
| **decision** | 技术决策、方案选择、权衡结果 | 决定、选择、采用、禁用 |
| **implementation** | 代码变更、配置修改、技术实施 | 修改、实现、配置、部署 |
| **issue** | 问题诊断、Bug 分析、根因识别 | Bug、问题、错误、失败 |
| **knowledge** | 领域知识、技术原理、最佳实践 | 原理、机制、特性、规范 |

## 1️⃣ Decision（决策）模板

### 标准结构

```markdown
Memory Name: [简洁的决策描述]
Memory Type: decision

Observation 1（决策内容，<100字）:
[明确说明做出的决策是什么]

Observation 2（决策原因，<100字）:
[解释为什么做出这个决策，关键考虑因素]

Observation 3（影响或后续，<100字）:
[说明决策的影响范围，或后续需要采取的行动]
```

### 示例 1：技术方案选择

```markdown
Memory Name: "采用 K-Planes 特征编码器增强密度预测"
Memory Type: decision
Metadata: {
  "project": "R²-Gaussian",
  "date": "2025-11-22",
  "author": "research_team"
}

Observations:
  1. "决定引入 K-Planes 编码器，使用三正交平面生成位置特征，增强 gaussian 密度预测的空间相关性。"（52字）
  2. "原因：稀疏视角下独立高斯缺乏空间关联，K-Planes 可利用邻域信息提升重建质量。"（42字）
  3. "需要修改 gaussian_model.py 和 kplanes.py，增加 encoder 训练分支。"（45字）
```

### 示例 2：参数调整决策

```markdown
Memory Name: "调整 K-Planes TV 正则化权重"
Memory Type: decision
Metadata: {
  "project": "R²-Gaussian",
  "technique": "SPAGS",
  "version": "v1"
}

Observations:
  1. "决定将 lambda_plane_tv 从 0.0002 调整为 0.002，增强特征平面平滑性。"（45字）
  2. "原因：原权重过小导致特征平面噪声过大，影响密度预测稳定性。"（33字）
  3. "验证：Foot 3views PSNR 提升 0.2dB，无明显过平滑现象。"（35字）
```

### Relations 示例

```python
# 决策通常连接到：
# - SOLVES: 解决的 issue
# - IMPLEMENTS: 实现此决策的 implementation
# - INFLUENCES: 影响的其他 decision

relations = [
  {"from": "decision_id", "to": "issue_id", "type": "SOLVES"},
  {"from": "implementation_id", "to": "decision_id", "type": "IMPLEMENTS"}
]
```

## 2️⃣ Implementation（实施）模板

### 标准结构

```markdown
Memory Name: [实施内容的简洁描述]
Memory Type: implementation

Observation 1（变更描述，<100字）:
[说明做了什么修改或实现了什么功能]

Observation 2（代码位置，<100字）:
[指明修改的文件和关键位置，使用格式：文件名:行号]

Observation 3（效果或验证，<100字）:
[说明实施后的效果，或如何验证]
```

### 示例 1：代码修改

```markdown
Memory Name: "FSGS 邻近密集化阈值调整"
Memory Type: implementation
Metadata: {
  "project": "R²-Gaussian",
  "files": ["proximity_densifier.py"],
  "lines": [89]
}

Observations:
  1. "将 proximity_threshold 默认值从 10.0 改为 5.0，限制邻近密集化的触发范围。"（48字）
  2. "修改位置：r2_gaussian/innovations/fsgs/proximity_densifier.py:89，参数 `threshold`。"（52字）
  3. "效果：高斯点数量稳定在 80k-120k，VRAM 占用降低 30%，训练稳定。"（40字）
```

### 示例 2：配置变更

```markdown
Memory Name: "SPAGS 训练超参数配置"
Memory Type: implementation
Metadata: {
  "project": "R²-Gaussian",
  "config_file": "run_spags_ablation.sh",
  "version": "v1"
}

Observations:
  1. "K-Planes 分辨率设为 64，特征维度设为 32，平衡精度和显存占用。"（38字）
  2. "TV 正则化权重设为 0.002，增强特征平面平滑性约束。"（30字）
  3. "FSGS 邻近阈值设为 5.0，k_neighbors 设为 5，控制密集化范围。"（35字）
```

### Relations 示例

```python
# implementation 通常连接到：
# - IMPLEMENTS: 实现的 decision
# - FIXES: 修复的 issue
# - VALIDATES_BY: 验证此实现的其他 implementation（如测试）

relations = [
  {"from": "implementation_id", "to": "decision_id", "type": "IMPLEMENTS"},
  {"from": "test_implementation_id", "to": "implementation_id", "type": "VALIDATES"}
]
```

## 3️⃣ Issue（问题）模板

### 标准结构

```markdown
Memory Name: [问题的简洁描述]
Memory Type: issue

Observation 1（问题现象，<100字）:
[描述观察到的问题表现，包括症状和发生场景]

Observation 2（根本原因，<100字）:
[解释问题的根本原因，技术层面的分析]

Observation 3（影响范围，<100字）:
[说明问题影响哪些功能或模块，严重程度]
```

### 示例 1：代码 Bug

```markdown
Memory Name: "FSGS 邻近密集化阈值过大导致 VRAM 溢出"
Memory Type: issue
Metadata: {
  "project": "R²-Gaussian",
  "severity": "critical",
  "discovered_date": "2025-11-22"
}

Observations:
  1. "训练 5000 步后，高斯点数量爆炸至 500k+，VRAM 占用超过 24GB，导致 OOM 崩溃。"（48字）
  2. "根因：proximity_threshold 默认值 10.0 过大，几乎所有区域都被判定为稀疏，触发过度密集化。"（52字）
  3. "影响：所有启用 FSGS 的训练任务在高迭代数时崩溃，需调整阈值至 3.0-7.0 范围。"（42字）
```

### 示例 2：性能问题

```markdown
Memory Name: "GR-Gaussian 训练内存占用过高"
Memory Type: issue
Metadata: {
  "project": "R²-Gaussian",
  "severity": "medium",
  "component": "GR-Gaussian"
}

Observations:
  1. "训练 GR-Gaussian 时，GPU 内存占用达到 22GB（A100-40GB 无法运行），相比 baseline 的 12GB 增加了 83%。"（58字）
  2. "根因：tri-plane 特征图尺寸过大（512×512×32），且同时存储 encoder 和 gaussian 梯度，导致峰值内存翻倍。"（58字）
  3. "影响：无法在单卡 A100-40GB 上训练，需要 A100-80GB 或多卡并行，增加了实验成本。"（44字）
```

### Relations 示例

```python
# issue 通常连接到：
# - DIAGNOSED_BY: 诊断此问题的 decision
# - FIXED_BY: 修复此问题的 implementation
# - CAUSED_BY: 引发此问题的其他 implementation

relations = [
  {"from": "decision_id", "to": "issue_id", "type": "DIAGNOSES"},
  {"from": "implementation_id", "to": "issue_id", "type": "FIXES"}
]
```

### Relations 示例

# - SUPPORTS: 支持的 decision（提供理论依据）
# - GUIDES: 指导的 implementation（提供实践指南）
# - RELATED_TO: 相关的其他 knowledge

relations = [
  {"from": "knowledge_id", "to": "decision_id", "type": "SUPPORTS"},
  {"from": "knowledge_id", "to": "implementation_id", "type": "GUIDES"}
]
```

## 🔗 关系类型参考

### 常用关系类型

| 关系类型 | 含义 | 典型用法 |
|---------|------|----------|
| **SOLVES** | 解决 | decision → issue |
| **IMPLEMENTS** | 实现 | implementation → decision |
| **FIXES** | 修复 | implementation → issue |
| **DIAGNOSES** | 诊断 | decision → issue |
| **VALIDATES** | 验证 | implementation → implementation |
| **SUPPORTS** | 支持 | knowledge → decision |
| **GUIDES** | 指导 | knowledge → implementation |
| **DEPENDS_ON** | 依赖 | implementation → implementation |
| **LEADS_TO** | 导致 | decision → decision |
| **CAUSED_BY** | 引起 | issue → implementation |
| **RELATED_TO** | 相关 | 任意 → 任意 |

### Relation 强度建议

- **1.0**：直接因果关系（如 IMPLEMENTS, FIXES）
- **0.8**：强相关（如 SOLVES, DIAGNOSES）
- **0.5**：中等相关（如 SUPPORTS, GUIDES）
- **0.3**：弱相关（如 RELATED_TO）

## ✅ 快速检查清单

在创建 memory 前，确认：

- [ ] Memory Name 简洁明确（≤ 20 字）
- [ ] Memory Type 正确（decision/implementation/issue/knowledge）
- [ ] 每个 observation < 150 字
- [ ] Observations 数量 ≤ 3 条
- [ ] 保持类型纯度（不混合多种类型）
- [ ] 原子化（可独立理解）
- [ ] 设置必要的 metadata（project, date, version 等）
- [ ] 添加 relations 连接相关 memory

        "author": "research_team"

    {"from": "kplanes_decision", "to": "previous_issue_id", "type": "SOLVES", "strength": 0.9}
  ]

```
