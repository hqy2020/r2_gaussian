# Neo4j Memory 记忆模板

本文档提供 4 种记忆类型的标准模板，确保记忆的颗粒度和结构符合最佳实践。

---

## 📋 模板使用指南

### 基本原则

1. **单个 observation < 150 字**（中文约 100 字）
2. **每个 memory 最多 3 个 observations**
3. **保持类型纯度**（不混合多种类型）
4. **原子化知识点**（可独立理解和复用）

### 何时使用哪种类型

| 类型 | 适用场景 | 关键词 |
|------|----------|--------|
| **decision** | 技术决策、方案选择、权衡结果 | 决定、选择、采用、禁用 |
| **implementation** | 代码变更、配置修改、技术实施 | 修改、实现、配置、部署 |
| **issue** | 问题诊断、Bug 分析、根因识别 | Bug、问题、错误、失败 |
| **knowledge** | 领域知识、技术原理、最佳实践 | 原理、机制、特性、规范 |

## 1️⃣ Decision（决策）模板

### 标准结构

```markdown
Memory Name: [简洁的决策描述]
Memory Type: decision

Observation 1（决策内容，<100字）:
[明确说明做出的决策是什么]

Observation 2（决策原因，<100字）:
[解释为什么做出这个决策，关键考虑因素]

Observation 3（影响或后续，<100字）:
[说明决策的影响范围，或后续需要采取的行动]
```

### 示例 1：技术方案选择

```markdown
Memory Name: "采用密度敏感的稀疏引导（SSS）机制"
Memory Type: decision
Metadata: {
  "project": "R²-Gaussian",
  "date": "2025-11-22",
  "author": "research_team"
}

Observations:
  1. "决定引入 SSS 机制，使用 tri-plane encoder 生成 density_offset，通过 sigmoid 调制 gaussian 不透明度。"（56字）
  2. "原因：稀疏视角下缺乏几何约束，SSS 可利用 CT density 信息引导 gaussian 密度分布，提升重建质量。"（48字）
  3. "需要修改 gaussian_model.py 和 render_query.py，增加 encoder 训练分支和 density-aware 渲染路径。"（51字）
```

### 示例 2：参数调整决策

```markdown
Memory Name: "禁用 Balance Loss L1 正则化"
Memory Type: decision
Metadata: {
  "project": "R²-Gaussian",
  "technique": "SSS",
  "version": "v3"
}

Observations:
  1. "决定暂时禁用 Balance Loss，将 opacity_reg_weight 从 0.01 改为 0.0。"（42字）
  2. "原因：在缺少 Student's t 渲染时，L1 正则化将所有 opacity 压缩到 0，破坏训练过程。"（43字）
  3. "后续：等 Student's t 渲染实现后再启用，或添加 opacity 下界约束（如 min=0.01）防止完全归零。"（49字）
```

### Relations 示例

```python
# 决策通常连接到：
# - SOLVES: 解决的 issue
# - IMPLEMENTS: 实现此决策的 implementation
# - INFLUENCES: 影响的其他 decision

relations = [
  {"from": "decision_id", "to": "issue_id", "type": "SOLVES"},
  {"from": "implementation_id", "to": "decision_id", "type": "IMPLEMENTS"}
]
```

## 2️⃣ Implementation（实施）模板

### 标准结构

```markdown
Memory Name: [实施内容的简洁描述]
Memory Type: implementation

Observation 1（变更描述，<100字）:
[说明做了什么修改或实现了什么功能]

Observation 2（代码位置，<100字）:
[指明修改的文件和关键位置，使用格式：文件名:行号]

Observation 3（效果或验证，<100字）:
[说明实施后的效果，或如何验证]
```

### 示例 1：代码修改

```markdown
Memory Name: "SSS sigmoid 调制公式修复"
Memory Type: implementation
Metadata: {
  "project": "R²-Gaussian",
  "files": ["gaussian_model.py"],
  "lines": [456]
}

Observations:
  1. "将 sigmoid 调制公式从 `sigmoid(density_offset)` 改为 `0.7 + 0.6 * sigmoid(density_offset)`，调制范围从 [0, 1] 扩展到 [0.7, 1.3]。"（79字）
  2. "修改位置：r2_gaussian/scene/gaussian_model.py:456，函数 `get_opacity_with_density_aware()`。"（52字）
  3. "效果：训练 5000 步后，opacity 保持在 [0.05, 0.95]，gaussian 分布稳定，无归零现象。"（44字）
```

### 示例 2：配置变更

```markdown
Memory Name: "SSS v3 训练超参数配置"
Memory Type: implementation
Metadata: {
  "project": "R²-Gaussian",
  "config_file": "train_sss_v3.sh",
  "version": "v3"
}

Observations:
  1. "Encoder 学习率设为 0.002，Decoder 学习率设为 0.001（0.5 倍），防止 decoder 过拟合。"（48字）
  2. "TV 正则化强度提升 10 倍：lambda_plane_tv 从 0.0002 增加到 0.002，增强平滑性约束。"（50字）
  3. "训练模式改为单模型（gaussiansN=1），与 baseline 保持一致，确保公平对比。"（39字）
```

### Relations 示例

```python
# implementation 通常连接到：
# - IMPLEMENTS: 实现的 decision
# - FIXES: 修复的 issue
# - VALIDATES_BY: 验证此实现的其他 implementation（如测试）

relations = [
  {"from": "implementation_id", "to": "decision_id", "type": "IMPLEMENTS"},
  {"from": "test_implementation_id", "to": "implementation_id", "type": "VALIDATES"}
]
```

## 3️⃣ Issue（问题）模板

### 标准结构

```markdown
Memory Name: [问题的简洁描述]
Memory Type: issue

Observation 1（问题现象，<100字）:
[描述观察到的问题表现，包括症状和发生场景]

Observation 2（根本原因，<100字）:
[解释问题的根本原因，技术层面的分析]

Observation 3（影响范围，<100字）:
[说明问题影响哪些功能或模块，严重程度]
```

### 示例 1：代码 Bug

```markdown
Memory Name: "SSS 机制 sigmoid 调制范围过窄导致训练失败"
Memory Type: issue
Metadata: {
  "project": "R²-Gaussian",
  "severity": "critical",
  "discovered_date": "2025-11-22"
}

Observations:
  1. "训练 2000 步后，所有 gaussian 的 opacity 归零（值为 0.000），导致渲染结果全黑，训练无法继续。"（50字）
  2. "根因：density_offset 直接作为 sigmoid 输入，调制范围 [0, 1]（仅 50%），不足以产生有效的 density-aware 效果，同时 L1 正则化将 opacity 压缩到 0。"（80字）
  3. "影响：所有使用 SSS 机制的训练任务失败，包括 Chest、Foot、Head 等 5 个器官数据集。"（42字）
```

### 示例 2：性能问题

```markdown
Memory Name: "GR-Gaussian 训练内存占用过高"
Memory Type: issue
Metadata: {
  "project": "R²-Gaussian",
  "severity": "medium",
  "component": "GR-Gaussian"
}

Observations:
  1. "训练 GR-Gaussian 时，GPU 内存占用达到 22GB（A100-40GB 无法运行），相比 baseline 的 12GB 增加了 83%。"（58字）
  2. "根因：tri-plane 特征图尺寸过大（512×512×32），且同时存储 encoder 和 gaussian 梯度，导致峰值内存翻倍。"（58字）
  3. "影响：无法在单卡 A100-40GB 上训练，需要 A100-80GB 或多卡并行，增加了实验成本。"（44字）
```

### Relations 示例

```python
# issue 通常连接到：
# - DIAGNOSED_BY: 诊断此问题的 decision
# - FIXED_BY: 修复此问题的 implementation
# - CAUSED_BY: 引发此问题的其他 implementation

relations = [
  {"from": "decision_id", "to": "issue_id", "type": "DIAGNOSES"},
  {"from": "implementation_id", "to": "issue_id", "type": "FIXES"}
]
```

### 标准结构

```markdown
Memory Name: [知识点的简洁描述]

Observation 1（核心内容，<100字）:

Observation 2（适用场景，<100字）:
[说明此知识在什么场景下适用，使用条件]

Observation 3（参考资源，<100字）:
[提供相关的文献、代码、文档链接等]
```

```markdown
Memory Name: "3D Gaussian Splatting 密度化与分裂机制"
Memory Type: knowledge

  "domain": "3DGS",
  "topic": "densification",
  "source": "3DGS Original Paper"
}

Observations:
  1. "密度化通过监控 view-space position gradient，当梯度超过阈值（如 0.0002）时，对大 gaussian 进行分裂（split），对小 gaussian 进行克隆（clone）。"（81字）
  2. "适用于：初始点云稀疏的场景，需要在训练过程中动态增加 gaussian 数量以覆盖未采样区域。每 100 步执行一次。"（56字）
  3. "参考：scene/gaussian_model.py:densify_and_split()，论文 3DGS Section 3.3 Adaptive Density Control。"（62字）
```

### 示例 2：最佳实践

```markdown

Metadata: {
  "domain": "Medical Imaging",
  "topic": "training",
  "project": "R²-Gaussian"

  1. "CT 重建任务推荐使用指数衰减学习率：初始 lr=0.002，每 1000 步衰减到 gamma 倍（gamma=0.95），最小 lr=0.0001。"（71字）
  2. "适用于：稀疏视角（3/6/9 views）CT 重建，可避免训练后期震荡，提升 PSNR 约 0.5-1.0 dB。"（48字）
  3. "参考：train.py:setup_lr_scheduler()，配置示例见 train_baseline.sh。"（45字）
```

```markdown
Memory Name: "R²-Gaussian 训练命名规范"
```

### Relations 示例

```python
# knowledge 通常连接到：
# - SUPPORTS: 支持的 decision（提供理论依据）
# - GUIDES: 指导的 implementation（提供实践指南）
# - RELATED_TO: 相关的其他 knowledge

relations = [
  {"from": "knowledge_id", "to": "decision_id", "type": "SUPPORTS"},
  {"from": "knowledge_id", "to": "implementation_id", "type": "GUIDES"}
]
```

## 🔗 关系类型参考

### 常用关系类型

| 关系类型 | 含义 | 典型用法 |
|---------|------|----------|
| **SOLVES** | 解决 | decision → issue |
| **IMPLEMENTS** | 实现 | implementation → decision |
| **FIXES** | 修复 | implementation → issue |
| **DIAGNOSES** | 诊断 | decision → issue |
| **VALIDATES** | 验证 | implementation → implementation |
| **SUPPORTS** | 支持 | knowledge → decision |
| **GUIDES** | 指导 | knowledge → implementation |
| **DEPENDS_ON** | 依赖 | implementation → implementation |
| **LEADS_TO** | 导致 | decision → decision |
| **CAUSED_BY** | 引起 | issue → implementation |
| **RELATED_TO** | 相关 | 任意 → 任意 |

### Relation 强度建议

- **1.0**：直接因果关系（如 IMPLEMENTS, FIXES）
- **0.8**：强相关（如 SOLVES, DIAGNOSES）
- **0.5**：中等相关（如 SUPPORTS, GUIDES）
- **0.3**：弱相关（如 RELATED_TO）

## ✅ 快速检查清单

在创建 memory 前，确认：

- [ ] Memory Name 简洁明确（≤ 20 字）
- [ ] Memory Type 正确（decision/implementation/issue/knowledge）
- [ ] 每个 observation < 150 字
- [ ] Observations 数量 ≤ 3 条
- [ ] 保持类型纯度（不混合多种类型）
- [ ] 原子化（可独立理解）
- [ ] 设置必要的 metadata（project, date, version 等）
- [ ] 添加 relations 连接相关 memory

```python
# 示例：记录一个决策
memory_store(
  memories=[
    {
      "localId": "sss_decision",
      "name": "采用密度敏感的稀疏引导（SSS）机制",
      "memoryType": "decision",
      "metadata": {
        "project": "R²-Gaussian",
        "date": "2025-11-22",
        "author": "research_team"
      },
      "observations": [
        "决定引入 SSS 机制，使用 tri-plane encoder 生成 density_offset，通过 sigmoid 调制 gaussian 不透明度。",
        "原因：稀疏视角下缺乏几何约束，SSS 可利用 CT density 信息引导 gaussian 密度分布，提升重建质量。",

    }
  ],
  relations=[
    {"from": "sss_decision", "to": "previous_issue_id", "type": "SOLVES", "strength": 0.9}
  ]
)
```
